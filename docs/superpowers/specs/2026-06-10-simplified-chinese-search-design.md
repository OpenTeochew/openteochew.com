# 簡體中文搜索支持

## 問題

系統底層全部使用繁體中文存儲（`han`、`mandarin` 等欄位）。用戶輸入簡體中文搜索時，LIKE 查詢無法匹配繁體內容，導致零結果。

例如：用戶搜「语」（簡體），但 DB 裡存的是「語」（繁體），搜不到。

## 目標

用戶輸入簡體中文也能搜到繁體內容，無需任何前端 UI 改動。分兩階段：

- **Phase 1**：後端搜索時自動將簡體轉繁體，最小改動、立刻見效
- **Phase 2**（不在本 spec 範圍）：全站簡/繁切換功能（UI 文字、查詢結果均支持簡體顯示）

## Phase 1：後端搜索轉換

### 方案

在後端 `searchEntries()` 中，對 `q_han` 和 `q_mandarin` 參數做簡→繁轉換，然後用轉換後的值進行 LIKE 查詢。

轉換在 SQL 查詢之前、參數構建之後進行。繁體輸入不受影響（簡→繁對繁體字是冪等的）。

### 適用欄位

| API 參數 | 轉換？ | 原因 |
|----------|--------|------|
| `q_han` | 是 | 漢字欄位，簡繁差異大 |
| `q_mandarin` | 是 | 普通話欄位，含中文字 |
| `q_puj` | 否 | 白話字（拉丁字母），無簡繁之分 |
| `q_dp` | 否 | 潮拼（拉丁字母） |
| `q_en` | 否 | 英文 |
| `q_ja` | 否 | 日文 |

### 庫選擇

使用 `opencc-js`（純 JS 實現 OpenCC 標準，無 native 依賴，兼容 Cloudflare Workers）。

- 子路徑導入 `opencc-js/cn2t`（只含簡→繁轉換表）
- Worker bundle 增加 ~1MB（原始），gzip 後 ~430KB，在 Workers 10MB 限制內
- Workers 有全局緩存，冷啟動影響可控

### 改動範圍

**只改一個文件**：`backend/src/server/services/search.ts`

```
searchEntries(db, params)
  ┌─ params.q_han = s2t(params.q_han)      // 新增
  ├─ params.q_mandarin = s2t(params.q_mandarin)  // 新增
  └─ 其餘邏輯不變（LIKE 查詢、分組、排序）
```

轉換函數只需初始化一次（模組級別），後續調用無開銷。

### 轉換細節

opencc-js 的 `CN2T` 轉換器使用 OpenCC 標準詞庫，覆蓋：
- 一對多映射（如「发」→「發/髮」根據詞組上下文）
- 常用字 3000+ 對應

對搜索場景，一對多不是問題——轉換結果作為 LIKE 模式使用，多出來的字元只會擴大匹配範圍（更寬鬆），不會漏掉結果。

### 邊界情況

| 情況 | 處理 |
|------|------|
| 輸入繁體 | 簡→繁冪等，結果不變 |
| 輸入混合簡繁 | 轉換只影響簡體部分，繁體部分不變 |
| 輸入非中文（英文、PUJ） | 轉換不影響拉丁字母 |
| `q_han` 為空 | 跳過轉換（現有邏輯已處理） |

## Phase 2 概覽（未來）

全站簡/繁切換的粗略思路，不在本次實現：

- opencc-js 在前端懶加載（~430KB gzip），僅用戶切換到簡體模式時載入
- Vue composable 提供繁→簡轉換，包裹 UI 文字和查詢結果
- 搜索輸入在前端做簡→繁轉換後發 API
- 用戶偏好存 localStorage

## 成功標準

- 輸入「语」能搜到含「語」的詞條
- 輸入繁體「語」結果與現在一致
- 無前端改動，無 DB 改動
